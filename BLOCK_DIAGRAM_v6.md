## КЛЮЧЕВЫЕ МОМЕНТЫ

### 1. Semantic Matching
- Токенизирует `filename` + `text[500 символов]`
- Находит пересечения с KB
- Передаёт ТОП-3 примера в LLM

### 2. CoT Reasoning
- Модель получает примеры
- Рассуждает пошагово (`reasoning_content`)
- Генерирует JSON

### 3. KB Override
- Если точное совпадение (`score > 0.90`)
- Подменяет `purpose`, `content_summary`, `document_type`

---

## Технические детали

### Этап 4: Semantic Matching

**Алгоритм:**
```python
def tokenize(text):
    return set(re.findall(r'[а-яА-Яa-zA-Z0-9]+', text.lower()))

file_tokens = tokenize(filename)           # ['пд', '9', 'мкд', '0109']
text_tokens = tokenize(doc_text[:500])     # ['админ', 'зарайск', '2025']
all_tokens = file_tokens.union(text_tokens)

for entry in kb_data:
    entry_tokens = tokenize(entry['title'] + entry['customer'])
    score = len(all_tokens.intersection(entry_tokens))
```

**Результат:**
- ТОП-3 записи с максимальным score
- Эти примеры передаются в промпт LLM

---

### Этап 6: LLM API

**Модель:** `mistralai/ministral-3-14b-reasoning`

**Параметры:**
- `max_tokens`: 8000
- `temperature`: 0.01
- `timeout`: 180 сек

**Возвращает:**
- `content`: JSON с 7 полями
- `reasoning_content`: пошаговый анализ (Chain-of-Thought)

---

### Этап 8: KB Override

**Условие:** `score > 0.90` (высокая уверенность в совпадении)

**Подменяемые поля:**
- `purpose` — цель документа
- `content_summary` — содержание
- `document_type` — тип документа

**Почему подменяем?**
Эти поля требуют детального знания контекста проекта. Если документ точно найден в KB, лучше взять готовое описание.
