# Работа с проектом Project Doc Analyzer

## Контекст проекта
- **Описание:** Веб-приложение для анализа проектной документации (PDF, DOCX, XML). Принимает документ, парсит его (в зависимости от формата используются разные методы извлечения) и возвращает метаданные в структурированном виде (поля: Название, Заказчик, Подрядчик, Год составления, Тип документа, Содержимое, Цель).
- **Стек:** Python 3.8+, Flask, PyMuPDF (fitz), python-docx, pytesseract (OCR).

## Ошибки
- [Неправильная интерпретация команды на форматирование и сохранение]
  Проблема: Пользователь попросил "сделать UUID формат" и "сохранить в (скриншот и код и md файлы)", что было понято превратно. Вместо запроса разъяснений, модель создала текстовый слепок скриншота и запустила скрипт изменения структуры папок (создав UUID-папки), что оказалось ошибкой.
  Как делать правильно: Если форматы (например, "UUID формат" или сохранение скриншота) не описаны явно в базе знаний проекта, необходимо сначала переспросить и запросить пример желаемого вывода. Не генерировать новые скрипты для изменения структуры папок без явного подтверждения.
  Дата: 2026-03-30 02:56

- [Ошибочный выбор LLM-модели для строгого парсинга JSON]
  Проблема: Модели с суффиксом `reasoning-distilled` (как qwen3.5-4b-claude-4.6-opus-reasoning-distilled) натренированы выдавать текстовые рассуждения (`self_check`, `confidence`), игнорируя строгий JSON-шаблон, что ломает работу автоматических парсеров.
  Как делать правильно: Для задач автоматического вывода JSON через API строго использовать официальные `Instruct` или `Coder` модели (к примеру, Qwen2.5-3B-Instruct), не имеющие надстроек "рассуждений".
  Дата: 2026-03-30 03:55
