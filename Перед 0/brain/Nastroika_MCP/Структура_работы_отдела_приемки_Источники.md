## Этап 2 — Ядро: RAG, PP963 и Внешние интеграции





### 2.1 Knowledge Base Agent (RAG)





- **Стек:** Pinecone/Weaviate + `text-embedding-3-large` (OpenAI) + `rerank-multilingual-v2.0` (Cohere).


- **Ключевая задача (`KnowledgeBaseIndexer`):** Реализация логики `smart_chunking`. Документы парсятся не по фиксированному количеству токенов, а по семантической структуре: глава -> пункт -> подпункт. Это предотвращает разрыв юридического контекста.





### 2.2 PP963ComplianceAgent





- Выполняет RAG-запросы и LLM-валидацию по пунктам чек-листа ПП №963 (16-20, 24 и др.).


- **Кросс-валидация ТЭП (Ground Truth Check):** Площадь в ПЗ (Раздел 1) vs АР (Раздел 3); этажность; реквизиты ГПЗУ.





### 2.3 External Integration Agent (Интеграция НОПРИЗ)





**Задача:** Автоматически верифицировать пп. 66 и 67 (членство ГИП в НОПРИЗ).





> [!WARNING]


> У НОПРИЗ нет публичного REST API. `https://nrs.nopriz.ru` — HTML-форма. Ниже описана реальная стратегия.





**Основная стратегия:** Playwright-парсер (`playwright.chromium.launch()`) — логин на `nrs.nopriz.ru`, заполнение формы поиска по ФИО + ИНН, парсинг HTTP-ответа. Сесссионные cookie сохраняются в `storage_state.json` для повторных запросов.





**Стратегия деградации при поломке парсера:** Задача автоматически передаётся `HumanInTheLoopAgent` с триггером `is_edge_case`. Эксперт проверяет реестр вручную и логирует результат в `Disagreement Log`.





**Мониторинг:** Добавить тест, который еженедельно проверяет корректность CSS-селекторов на странице НОПРИЗ (автоматический алерт при изменении вёрстки).

## Этап 3 — Масштабирование: Схемы теплоснабжения (ПП №154)





### 3.1 Разработка `PP154ComplianceAgent`





**Это полноценный, сложный домен экспертизы, требующий отдельного спринта:**


- Проверка обязательного наличия **13 разделов** схемы теплоснабжения.


- Математическая валидация баланса тепловой энергии (Производство = Потребление + Потери). Допустимый коридор невязки: **≤ 2%**.


- Проверка горизонта планирования инвестиционной программы: **≥ 15 лет**.


- Контроль наличия файлов "Электронной модели системы теплоснабжения" (валидация специфичных расширений ПО: Zulu Thermo, Пирамида).

## Этап 4 — Результаты: Заключение и Human-in-the-Loop





### 4.1 Report Generator Agent





- **Сборка PDF (reportlab):** Формирование отчёта строго по ГОСТ Р 7.0.97-2016 (шрифты, отступы, нумерация).


- **Двумерная группировка:** 


1. По разделам (`by_section`: ПЗ, АР, КР, ООС...).


2. По критичности (`by_severity`: critical / warning / info).





### 4.2 Human-in-the-Loop Agent





**Полная реализация 4-х триггеров перехвата управления:**


1. `critical_error`: Выявлено критическое несоответствие (например, нет УИН или ЭЦП/ИУЛ).


2. `confidence < 0.7`: Уверенность LLM или RAG ниже 70%.


3. `agent_disagreement`: Оркестратор фиксирует конфликт выводов меджу агентами (например, DocumentAnalyzer говорит "ЭЦП есть", а PP963Agent говорит "Документ не подписан").


4. `is_edge_case`: Распознан нестандартный или редкий тип объекта (например, атомная станция).





**Задача:** Логирование ответов эксперта в `Disagreement Log` для дообучения.

Но как ИИ-архитектор, я обязан выдать жесткую критику по формату данных (именно по функции парсинга в тексте). Если мы оставим парсинг в текущем виде, наши умные LLM начнут ошибаться и галлюцинировать.





Вот 3 архитектурные проблемы краулера и способы их решения:





1. Сохранение в raw_text.txt убьет структуру (Критично)


В функции parse_cntd вы делаете content_el.get_text(separator="\n", strip=True). Это превращает размеченный закон (где есть Заголовки, Статьи, Пункты 1.1, Подпункты "а") в сплошную "простыню" текста. Почему это убьет RAG: Когда векторная база будет "нарезать" 100-страничный СП на куски (чанки) по 1000 символов, она разрежет текст посередине. ИИ получит кусок текста с правилом, но не будет знать, к какой Главе или Группе зданий это правило относится (потому что заголовок остался в прошлом чанке). ✅ Решение: Сохранять документы не в .txt, а в .md (Markdown). Нужно использовать библиотеку типа markdownify. Она сохранит заголовки как # Глава 1, а пункты как маркированные списки. Тогда "умный чанкер" (Smart Chunking) сможет нарезать текст математически точно по заголовкам (#, ##).





2. Полное уничтожение Таблиц в СП и ГОСТах (Критично)


Своды Правил (например, СП 42.13330 или таблицы огнестойкости в ФЗ-123) на 30% состоят из таблиц (расстояния, классы, коэффициенты). Метод .get_text() превращает HTML-таблицу в бессмысленный столбик терминов и цифр, теряя связь строк и столбцов. Агент PP963 просто не сможет вытащить нужное расстояние от садика до котельной. ✅ Решение: Перед парсингом текста нужно найти все теги в HTML и программно сконвертировать их в формат таблиц Markdown (| Колонка 1 | Колонка 2 |). Только в таком виде LLM способна "понимать" таблицы.





3. Защита от ботов на docs.cntd.ru (Риск)


Вы написали: "Если будет 403 — запустим через Playwright". CNTD (Кодекс) славится очень жесткой защитой от парсинга (Cloudflare/Qrator). Обычный aiohttp даже с заголовками от Chrome, скорее всего, отвалится с 403 ошибкой на 10-м документе. ✅ Решение: Если aiohttp будет падать, нам действительно придется перевести краулер на Playwright не как на запасной, а как на основной вариант загрузки "сложных" страниц, где нужно прождать JS-проверку браузера 5 секунд (в Playwright это page.wait_for_selector('.document-content')).

⚠️ Три реальные проблемы в стеке


1. Jinja2 + reportlab — они не работают вместе напрямую


Jinja2 — шаблонизатор HTML/текста. reportlab — программная генерация PDF через Python-объекты. Это разные парадигмы:





Либо только reportlab (код строит PDF напрямую — как в 





.docx


)


Либо Jinja2 → HTML → weasyprint (рендер HTML в PDF)


Выбери одно. Для строгого ГОСТ Р 7.0.97-2016 (поля, шрифты, нумерация) — reportlab надёжнее, он уже описан в документации.





2. spaCy для NER на русском — нужна конкретная модель


В плане написано просто "spaCy + custom NER". Для русского нужен либо ru_core_news_lg (базовый), либо дообученная модель под реквизиты (ФИО ГИПа, ИНН, номер ГПЗУ). Это отдельная задача, которой нет в спринтах — либо вписать в Спринт 5, либо заменить NER на LLM-промпт с structured output (проще и точнее для реквизитов).





3. SQLite / PostgreSQL — нужно выбрать сразу


Миграция SQLite → PostgreSQL в середине разработки болезненна. Для MVP с асинхронным FastAPI — сразу PostgreSQL + asyncpg. SQLite не дружит с async-записью.





💡 Одно недостающее звено


Шаг 4 — стратегия деградации при падении НОПРИЗ есть (is_edge_case → Human-in-the-Loop), но нет кеша результатов. Один и тот же ГИП проверяется при каждой подаче заново. Простое решение: хранить результат проверки ФИО+ИНН с TTL = 7 дней в Redis или в той же PostgreSQL-таблице.

## Этап 2 — Ядро: RAG, Агенты соответствия и Внешние интеграции





### 2.1 Knowledge Base Agent (RAG)





- **Стек:** Pinecone/Weaviate + `text-embedding-3-large` (OpenAI) + Cohere Rerank


- **Источники для индексации:** ПП №963 (актуальная редакция), ГрК РФ, ФЗ №190, ФЗ №116, 50+ СП, 30+ ГОСТ


- **Размер базы:** 500 000+ чанков (чанк = 512 токенов с перекрытием 50 токенов)


- **Запрос:** гибридный поиск BM25 (точный) + cosine similarity (семантический), Reranker на топ-20 → топ-5





### 2.2 PP963ComplianceAgent





- Принимает JSON от `DocumentAnalyzerAgent`


- Для каждого пункта чек-листа (16-20, 24-28...) формирует RAG-запрос → получает норму из КБ → через LLM (GPT-4 Turbo / Claude 3.5) верифицирует соответствие


- **Ground Truth Check — кросс-валидация ТЭП:**


- Общая площадь из XML Раздела 1 (ПЗ) vs. XML Раздела 3 (АР) — допустимое отклонение < 1м²


- Этажность из ПЗ vs. графической части (АР)


- Реквизиты ГПЗУ в ПЗ vs. приложенный файл ГПЗУ





### 2.3 External Integration Agent (НОВЫЙ — интеграция НОПРИЗ)





**Задача:** Автоматически верифицировать пп. 66 и 67 (членство ГИП в НОПРИЗ).





**Входные данные:** `gip_fio`, `gip_inn` из JSON DocumentAnalyzerAgent





**Алгоритм:**


1. Запрос к API НОПРИЗ (`https://nrs.nopriz.ru`) по ФИО и ИНН ГИПа


2. Парсинг ответа (статус: «Действующий» / «Исключён»)


3. Запись результата в JSON: `{"66": "PASS/FAIL", "67": "PASS/FAIL"}`





**Резервный вариант (если API нет):** Playwright-парсинг веб-страницы реестра с сохранением сессии.
